# Strategy Deck

Create compelling strategy presentations that synthesize discovery findings into actionable recommendations.

**Typical audience:** Leadership team, steering committee
**Presentation time:** 30-60 minutes
**Slide count:** 15-25 slides

---

## Deck Structure

### Standard Strategy Deck (20 slides)

| Section | Slides | Purpose |
|---------|--------|---------|
| **Opening** | 1-3 | Context, agenda, executive summary |
| **Current State** | 4-7 | Challenges, landscape, findings |
| **Insights** | 8-11 | Patterns, learnings, stakeholder themes |
| **Recommendations** | 12-16 | Prioritized solutions, quick wins |
| **Roadmap** | 17-19 | Phased plan, timeline, milestones |
| **Close** | 20 | Next steps, ask, discussion |

### Section Details

**Section 1: Opening (Slides 1-3)**
- Slide 1: Title (engagement name, client, date)
- Slide 2: Agenda (4-6 items max)
- Slide 3: Executive Summary (standalone ‚Äî execs may only see this)

**Section 2: Current State (Slides 4-7)**
- Slide 4: Engagement Context (why we're here, scope, sessions conducted)
- Slide 5: Challenge Landscape (top challenges, impact indicators)
- Slides 6-7: Key Findings by Dimension (People, Process, Technology)

**Section 3: Insights (Slides 8-11)**
- Slide 8: Pattern Summary (3-4 dominant patterns)
- Slides 9-10: Deep Dive on Top Patterns
- Slide 11: Stakeholder Alignment (champions vs. skeptics)

**Section 4: Recommendations (Slides 12-16)**
- Slide 12: Recommendation Overview (Quick Wins, Strategic, Future)
- Slide 13: Quick Wins Detail
- Slides 14-15: Strategic Recommendations (one per slide)
- Slide 16: Solutions Summary (challenge ‚Üí solution mapping)

**Section 5: Roadmap (Slides 17-19)**
- Slide 17: Phased Approach (Phase 1-3 timeline)
- Slide 18: Phase 1 Detail (initiatives, owners, metrics)
- Slide 19: Investment & Returns

**Section 6: Close (Slide 20)**
- Next steps, decisions needed, open questions, contact info

## Content Guidelines

### Writing for Slides

**Headlines:**
- Make headlines complete thoughts, not labels
- Bad: "Challenges"
- Good: "Manual processes consume 20+ hours per week"

**Bullet Points:**
- Maximum 5 bullets per slide
- Each bullet: 1-2 lines max
- Start with action verbs or key nouns

**Data Presentation:**
- Round numbers for impact (20+ hours, not 21.3 hours)
- Use comparisons (3x faster, 50% reduction)
- Cite sources (stakeholder quotes, session dates)

### Evidence Standards

| Confidence | How to Present |
|------------|----------------|
| High (direct quote) | "Stakeholder said: '[quote]'" |
| Medium (inference) | "Multiple stakeholders indicated..." |
| Low (interpretation) | "Initial signals suggest..." (flag for validation) |

### Avoiding Common Mistakes

| Mistake | Fix |
|---------|-----|
| Too much text | One idea per slide, use speaker notes |
| Jargon overload | Use client's terminology |
| No "so what" | Every finding needs an implication |
| Missing evidence | Support claims with discovery data |

## Deck Generation Process

1. **Gather Content:**
   - Run synthesis ‚Üí Get insights, patterns, prioritized challenges/solutions
   - Query data ‚Üí Get client details, stakeholders, session history

2. **Structure Content:**
   - Map insights to slide structure
   - Select top 3-4 patterns for deep dive
   - Choose 3-5 recommendations to feature

3. **Generate Slides:**
   - Read pptx skill for creation workflow
   - Apply Cadre brand colors and typography
   - Create HTML for each slide, convert to PPTX

4. **Quality Check:**
   - [ ] Executive summary tells complete story
   - [ ] Every insight has supporting evidence
   - [ ] Recommendations link to specific challenges
   - [ ] Roadmap is realistic and sequenced

## Exec Content Guidelines

### Writing Style

- Confident and direct
- Evidence-based but not academic
- Business-focused, not technical
- Active voice throughout
- Short sentences (15-20 words average)

### What to Include vs. Exclude

| Include | Exclude |
|---------|---------|
| Business impact | Technical details |
| Quantified findings | Methodology description |
| Prioritized recommendations | All possible options |
| Specific next steps | Background on Cadre |

### Headline Writing

| Weak Headline | Strong Headline |
|---------------|-----------------|
| "Data Challenges" | "Manual data handling costs 20+ hours weekly" |
| "Technology Findings" | "Four disconnected systems create data silos" |
| "Recommendation" | "Automated reporting delivers quick win in 2 weeks" |

### Length Constraints

| Section | Max Length |
|---------|------------|
| Context | 3 sentences |
| Each finding | 2 sentences |
| Primary insight | 3 sentences |
| Each recommendation | 1 sentence + outcome |
| Each next step | 1 line |

## Exec Generation Process

1. **Gather Content:**
   - Run synthesis ‚Üí Get top 5-7 insights, patterns, recommendations
   - Query data ‚Üí Get client name, scope, session count

2. **Filter and Prioritize:**
   - Select top 4-5 findings (high confidence only)
   - Choose 1 primary insight (most important "so what")
   - Pick 3-4 recommendations (immediate, near-term, strategic)

3. **Draft Content:**
   - Write headlines first (complete thoughts)
   - Add one sentence context per finding
   - Write primary insight paragraph (make it quotable)

4. **Quality Check:**
   - [ ] Fits on 1 page (2 max)
   - [ ] Can be understood in 3 minutes
   - [ ] Headlines tell the story on their own
   - [ ] Primary insight is quotable
   - [ ] Recommendations are actionable

# Findings Report

Create comprehensive documentation that captures all discovery findings, analysis, and recommendations in detail.

**Typical audience:** Project teams, technical stakeholders
**Reading context:** Deep dive, reference, archive
**Length:** 10-25 pages

---

## Report Structure

### Standard Structure (15-20 pages)

```
1. Executive Summary (1 page)
2. Engagement Overview (1-2 pages)
3. Methodology (1 page)
4. Findings by Dimension (6-10 pages)
   - People
   - Process
   - Technology
   - Challenges
   - Solutions
5. Pattern Analysis (2-3 pages)
6. Recommendations (2-3 pages)
7. Roadmap Summary (1 page)
8. Appendices (as needed)
```

### Section Purposes

| Section | Purpose | Key Content |
|---------|---------|-------------|
| Executive Summary | Standalone overview | Key findings, recommendations |
| Engagement Overview | Context setting | Scope, objectives, stakeholders |
| Methodology | Credibility | Approach, framework, data sources |
| Findings | Evidence | What we found, by dimension |
| Patterns | Analysis | Themes, correlations, insights |
| Recommendations | Action | Prioritized solutions, rationale |
| Roadmap | Planning | Phased approach, timeline |
| Appendices | Reference | Full data, lists |

> Multiple stakeholders indicated [finding]. (Sessions: [Date], [Date])

**Document references:**
> Per [Document Name], [specific data or observation].

### Confidence Calibration

| Confidence | Evidence Standard | Presentation |
|------------|-------------------|--------------|
| High | 3+ sources, direct evidence | State as fact |
| Medium | 2 sources, reasonable inference | "Evidence suggests..." |
| Low | Single source, interpretation | "Initial signals indicate..." |

## Report Generation Process

1. **Gather All Data:**
   - Query data ‚Üí All stakeholders, sessions, findings, challenges, solutions

2. **Structure Content:**
   - Group findings by dimension
   - Order by importance within each
   - Select patterns for analysis section

3. **Draft Report:**
   - Write executive summary last
   - Include evidence for every finding
   - Ensure recommendations link to findings

4. **Quality Check:**
   - [ ] Every finding has evidence cited
   - [ ] Confidence levels calibrated correctly
   - [ ] Patterns supported by evidence count
   - [ ] Recommendations link to specific findings

# Implementation Roadmap

Create phased action plans that sequence recommendations into executable timelines.

**Typical audience:** Project sponsors, implementation teams
**Timeframes:** Usually 3-12 months
**Formats:** Document section, standalone visual, or slide(s)

---

## Roadmap Structure

### Standard Three-Phase Model

| Phase | Name | Typical Duration | Focus |
|-------|------|------------------|-------|
| 1 | Quick Wins | 2-4 weeks | Build credibility, early ROI |
| 2 | Foundation | 4-12 weeks | Core capabilities, integrations |
| 3 | Scale | 3-6 months | Optimization, expansion |

### Phase Purposes

**Phase 1: Quick Wins**
- Demonstrate value fast
- Build organizational confidence
- Address "skeptic" stakeholders
- Low risk, high visibility

**Phase 2: Foundation**
- Establish core infrastructure
- Complete key integrations
- Build team capabilities
- Formalize processes

**Phase 3: Scale**
- Expand successful pilots
- Optimize based on learnings
- Add advanced capabilities
- Measure and refine ROI

---

## Phase Design

### Selection Criteria

**Phase 1 (Quick Wins):**
- Impact = High AND Effort = Low
- Minimal dependencies
- Visible results in <4 weeks
- Strong stakeholder support

**Phase 2 (Foundation):**
- Score ‚â• 12 from prioritizer
- Dependencies from Phase 1 resolved
- Moderate complexity acceptable
- 4-12 week implementation

**Phase 3 (Scale):**
- Strategic importance
- Builds on Phase 1-2 success
- Higher complexity acceptable
- Clear ROI model

### Sequencing Rules

1. **Dependencies first** ‚Äî If B requires A, schedule A before B
2. **Quick wins early** ‚Äî Build credibility before big bets
3. **Root causes before symptoms** ‚Äî Fix foundations first
4. **Stakeholder alignment** ‚Äî Consider change readiness
5. **Resource constraints** ‚Äî Don't overload teams

### Content Elements

| Element | Description |
|---------|-------------|
| **Name** | Descriptive label (not just "Phase 1") |
| **Duration** | Specific timeframe |
| **Objectives** | 2-3 bullets on what success looks like |
| **Initiatives** | Specific projects/workstreams |
| **Milestone** | Concrete deliverable marking completion |
| **Dependencies** | What must be true to start |
| **Success Metrics** | How we measure completion |
| **Risks** | Key risks and mitigations |

### Milestones

Milestones should be concrete, verifiable, meaningful, and celebratable.

| Weak Milestone | Strong Milestone |
|----------------|------------------|
| "Phase 1 complete" | "Daily reports delivered automatically by 8am" |
| "Integration done" | "CRM and ERP sync in real-time with <1% error rate" |
| "Team trained" | "All dispatchers certified on new scheduling tool" |

# Solution Brief

Complete reference for workshopping solution briefs with users. Combines workshop protocol, question framework, and module templates.

---

## Brief Overview

The Solution Brief is a single evolving document that grows from discovery through implementation.

**Triggers:**
- `/brief [solution name]` ‚Äî Standalone brief for any solution
- `/solutions brief [use case]` ‚Äî Brief for AI solution from discovery pipeline

### Module Structure

| Module | Name | Owner | Milestone |
|--------|------|-------|-----------|
| **0** | Discovery & Validation | Strategist | Feasibility confirmed |
| **1** | Problem & Context | Strategist | Quick pitch ready |
| **2** | Solution Concept | Strategist | Stakeholder alignment ready |
| **3** | Constraints & Data | Strategist | Technical feasibility validated |
| **4** | Technical Approach | Engineer | Engineering handoff ready |
| **5** | Estimates & ROI | Engineer | Investment decision ready |
| **6** | Risks & Adoption | Collaborative | Implementation ready |

Users can stop at any module. Module 1 alone = quick pitch. Modules 0-3 = stakeholder alignment. All modules = implementation-ready.

---

## Workshop Protocol

### Session Opening

Adapt opening based on what user provides:

**Scenario A: Solution name only**
1. Acknowledge the solution
2. Ask if they want to search project knowledge for context
3. Present assessment or ask what they'd like to work on

**Scenario B: Solution name + context**
1. Search for referenced context
2. Present what you found with citations
3. Show module status, propose which to work on

**Scenario C: Existing document provided**
1. Analyze provided content
2. Map content to modules
3. Show status, propose next module

### Status Display

After assessment, always show module status:

```
## Brief Status: [Solution Name]

| Module | Status | Notes |
|--------|--------|-------|
| 0. Discovery | ‚úÖ Complete | Validated in Oct 15 call |
| 1. Problem | ‚úÖ Complete | ‚Äî |
| 2. Solution | üîÑ In Progress | Scope needs refinement |
| 3. Constraints | ‚¨ú Not Started | ‚Äî |

**Recommendation:** Let's finish Module 2 ‚Äî specifically the scope boundaries.
```

### The Core Loop

For each module:
1. **Search** (if user opted in) ‚Äî Find relevant context
2. **Synthesize** ‚Äî Draft content based on findings
3. **Present** ‚Äî Show draft with citations
4. **Discuss** ‚Äî Get feedback
5. **Refine** ‚Äî Update based on feedback
6. **Confirm** ‚Äî Get approval to mark complete

### Generate Draft, Then Discuss

Default to drafting content rather than asking endless questions.

**Do this:**
```
Based on the Oct 15 discovery call, here's my draft for the Problem section:

**Problem Statement**
The rebate team spends 4-6 hours researching utility programs for each project.
(Source: Fireflies - Oct 15 Discovery, 23:45)

Does this capture the problem accurately? Anything to add?
```

**Not this:**
```
What's the problem you're trying to solve?
Who experiences this problem?
How much time do they spend on it?
```
